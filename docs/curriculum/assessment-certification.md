# Assessment and Certification

## Assessment layers

### Knowledge checks

Purpose: confirm understanding inside a lesson.

- Low stakes.
- Immediate explanation.
- Unlimited retry.
- No certificate evidence by itself.

### Module quizzes

Purpose: test concept selection and diagnosis across a module.

- Versioned question bank.
- Passing threshold set by module.
- Explanations after submission.
- Attempt history retained.
- Randomization cannot change difficulty materially.

### Guided labs

Purpose: apply the method with support.

- Hints and checklists allowed.
- Deterministic readiness checks.
- Revision expected.
- Evidence may support Aware or Guided mastery.

### Independent assessments

Purpose: demonstrate normal work without step-by-step support.

- Fixed time or attempt limits when appropriate.
- No answer-revealing hints.
- Version-pinned scenario, rubric, and model configuration.
- Evidence may support Independent mastery.

### Capstones

Purpose: integrate several competencies under ambiguity.

- Mixed inputs and realistic constraints.
- Required work log, QA, and reflection.
- Automated checks prepare a review packet.
- Human moderation required for launch certificates.

## Evaluation packet

Every assessed submission provides:

- Learner and entitlement state.
- Scenario and source-pack versions.
- Attempt and run history.
- Final artifact.
- Prompt and workflow versions when relevant.
- Deterministic validation results.
- Rubric scores and evidence snippets.
- Safety and policy flags.
- AI configuration and usage metadata.
- Learner reflection.
- Reviewer decision and comments.

## Reviewer decisions

- `APPROVE`: evidence satisfies the requirement.
- `APPROVE_WITH_NOTE`: requirement satisfied; improvement note remains.
- `REVISION_REQUIRED`: specific criteria must be corrected.
- `REJECT_POLICY`: submission violates integrity, data, or safety rules.
- `ESCALATE`: second reviewer or admin judgment required.

## Certification rules

A course certificate requires:

- Active eligible enrollment.
- All required modules and assessments complete.
- Minimum competency levels met.
- Required capstone evidence approved.
- No unresolved integrity or policy case.
- Certificate requirement version recorded.

The certificate stores:

- Learner display name.
- Course and version.
- Requirement version.
- Evidence identifiers.
- Reviewer or approved rule identifier.
- Issue date.
- Verification code.
- Status and status history.

## Verification page

Public verification displays:

- Learner-approved display name.
- Certificate title.
- Issue date.
- Active, expired, or revoked status.
- Course outcomes.
- Selected Skills Passport link when public.

It never exposes private submissions, raw prompts, source files, workspace membership, payment data, or internal reviewer notes.

## Integrity controls

- Synthetic scenario variables rotate within calibrated bounds.
- Duplicate artifact patterns create a review signal, not automatic guilt.
- The learner must explain key decisions.
- Work logs and revision history provide process evidence.
- A model score cannot be the sole capstone decision.
- Live-test cheating requests are refused.
- Certificates can be revoked with a reason and audit event.

## Appeals

1. Learner requests review within the published window.
2. A different reviewer receives the full packet and first decision.
3. Second reviewer scores against the same rubric version.
4. Material rubric ambiguity is escalated to curriculum governance.
5. Decision and rationale enter the audit history.
